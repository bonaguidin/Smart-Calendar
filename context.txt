# Smart Calendar Project Overview

## Project Structure
The project follows a client-server architecture with React frontend and Flask backend.

## Backend Components

### Main Server [main.py]
- Core Flask server implementation
- Handles HTTP endpoints for task management
- Implements task CRUD operations
- Manages task scheduling and recurrence
- Provides API endpoints for task parsing and editing
- Handles conversational intents and responses through intent detection
- Routes conversational queries to LLM for dynamic responses
- Integrates with PostgreSQL database for persistent storage
- Implements user authentication with JWT
- Provides location-based API endpoints for weather and traffic
- Uses Flask-Migrate for database schema management

Key Functions:
- `find_relevant_task(user_query, tasks_dict)`: Matches tasks based on user queries with relevance scoring
- `parse_task()`: Handles task parsing and conversational responses
- `add_task()`: Creates new tasks with timezone awareness
- `update_task_by_group_id(group_id)`: Updates all instances of a task
- `delete_task_group(group_id)`: Removes all instances of a task
- `create_recurring_instances(task)`: Generates instances for recurring tasks
- `calculate_next_occurrence(current_date, recurrence_pattern)`: Determines next task occurrence
- `get_weather(current_user)`: Retrieves weather forecast for a location
- `get_traffic(current_user)`: Gets traffic info between locations
- `search_location(current_user)`: Searches for location coordinates
- `get_task_location_data(current_user, task_id)`: Gets location data for a specific task

### Database Configuration [database.py]
- Manages PostgreSQL database connection
- Handles SQLAlchemy configuration
- Implements database migration support with Flask-Migrate
- Provides graceful error handling for database operations
- Supports development and production database configurations

Key Functions:
- `init_db(app)`: Initializes database connection with app context
- `db`: SQLAlchemy instance for database operations

### Database Migrations [migrations/]
- Alembic migration scripts
- Version control for database schema
- Supports schema evolution
- Enables safe database updates
- Tracks schema change history

Key Files:
- `env.py`: Alembic environment configuration
- `script.py.mako`: Migration script template
- `versions/`: Individual migration scripts

### App Configuration [app.py]
- Exposes Flask application for migrations
- Simplifies Flask-Migrate configuration
- Prevents duplicate SQLAlchemy initialization
- Supports CLI commands for database operations

### Models [models.py]
- Defines SQLAlchemy data models
- Implements User, Task, and Preferences models
- Provides data validation methods
- Handles database relationships
- Contains serialization methods for API responses

Key Models:
- `User`: User profile with authentication and preferences
- `UserPreference`: User settings for UI and features
- `Task`: Calendar tasks with recurrence and location support

### Authentication [auth.py]
- Implements JWT-based authentication
- Handles user registration and login
- Manages user profile operations
- Provides middleware for protected routes
- Secures API endpoints with token verification

Key Functions:
- `create_token(user_id, expires_in)`: Creates JWT token for authentication
- `token_required`: Decorator for protected routes
- `register()`: Creates new user accounts
- `login()`: Authenticates users and issues tokens
- `get_profile()`: Retrieves user profile data
- `update_profile()`: Updates user preferences
- `change_password()`: Updates user password securely
- `logout()`: Handles user session termination

### Location Services [location_services.py]
- Provides geocoding for addresses
- Integrates with weather forecast APIs
- Implements traffic and routing functionality
- Caches responses for improved performance
- Handles API error conditions gracefully

Key Functions:
- `geocode_location(location_query)`: Converts location string to coordinates
- `get_weather_forecast(location, days)`: Retrieves weather data
- `get_traffic_estimate(origin, destination)`: Gets route and traffic info

### Intent Detector [intent_detector.py]
- Detects user intents from natural language inputs
- Classifies inputs as conversational or task-related
- Routes conversational intents to LLM for dynamic responses
- Handles various conversational patterns

Key Functions:
- `detect_intent(user_input)`: Determines if input is conversational or task-related

### LLM Handler [llm_handler.py]
- Manages interactions with the language model
- Processes natural language task inputs
- Handles task parsing and edit requests
- Implements structured response generation
- Generates dynamic conversational responses
- Uses hybrid approach with local regex parsing for simple tasks
- Maintains proper separation between task and conversational responses

Key Functions:
- `create_openai_client()`: Initializes OpenAI client with SambaNova configuration
- `generate_structured_response(prompt_template, schema)`: Generates optimized validated LLM responses
- `parse_tasks_from_input(user_input, start_date)`: Uses hybrid approach - local parsing for simple cases, LLM for complex ones
- `parse_user_input(user_input, start_date)`: Local regex-based parser for common task patterns
- `parse_edit_query(user_input, original, context)`: Processes edit requests with optimized prompts
- `is_edit_request(user_input)`: Detects if input is requesting task modification
- `generate_conversational_response(user_input)`: Creates concise conversational responses with proper is_conversational flag

### Task Editor [task_editor.py]
- Handles task modification operations
- Manages task update logic
- Implements task validation
- Handles recurring task patterns

Key Classes:
- `EditRequest`: Dataclass for edit request handling
- `EditResult`: Dataclass for edit operation results
- `TaskEditor`: Main class for task editing operations

Key Functions:
- `process_edit(request)`: Processes edit requests with validation
- `parse_edit_request(input_text, original_task)`: Parses natural language edits
- `_handle_absolute_time(time_str, original_task)`: Processes absolute time changes
- `_handle_relative_time(direction, amount, unit, original_task)`: Handles relative time shifts
- `_validate_changes(edit_ops, original_task)`: Validates proposed changes

### JSON Validator [json_validator.py]
- Validates JSON responses from LLM
- Ensures data structure consistency
- Implements schema validation with minimal overhead
- Handles error formatting
- Optimized for token efficiency

Key Methods:
- `extract_json(raw)`: Efficiently extracts JSON from text using targeted regex
- `validate_data(data, schema)`: Performs lightweight schema validation
- `clean_and_validate(raw_text, schema, context)`: Streamlined validation process

### Timezone Handler [timezone_handler.py]
- Manages timezone conversions
- Handles date/time formatting
- Provides timezone-aware operations
- Ensures consistent date handling

### Schemas [schemas.py]
- Defines JSON schemas for task data
- Specifies validation rules
- Defines required fields and types
- Maintains data structure consistency

### Task Matcher [task_matcher.py]
- Implements fuzzy matching for tasks
- Handles task search and identification
- Provides relevance scoring
- Supports edit request matching

### Edit Detector [edit_detector.py]
- Detects edit intentions in user input
- Identifies edit operations
- Classifies edit types
- Supports natural language processing

### Edit History [edit_history.py]
- Tracks task modification history
- Maintains audit trail
- Supports undo/redo operations
- Records change metadata

#### Database Integration Updates

The following functions have been updated to work with database models instead of in-memory dictionaries:

- `create_recurring_instances(task)`: Now creates Task model instances for recurring tasks and returns them as a list.
  - Accepts a Task model instance as input rather than a dictionary
  - Creates new Task objects with the same attributes as the base task but with different dates
  - Returns a list of created instances that can be added to the database session
  - Includes proper logging and error handling

- The implementation now handles:
  - Skipping the first occurrence (which is already covered by the base task)
  - Creating appropriate recurring instances based on the recurrence pattern
  - Setting appropriate metadata like group_id, day_number, and total_days
  - Respecting the recurrence end date with a default of 3 months if not specified

This allows for proper database-driven recurring task management and completes the PostgreSQL integration for the tasks functionality.
